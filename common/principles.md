# 信息架构原则（P1-P10）与实践指南

> **职责**：理解 Plugin 信息架构设计决策的元参考 + 组件开发基础实践原则。按需加载——需要理解设计理念或做架构决策时阅读。

## 为什么是信息架构

Plugin "代码"是 Markdown，"运行时"是 LLM 上下文窗口。关键差异：上下文窗口是**硬约束**（超出即丢失），路由靠 description **语义匹配**（可能误触发），LLM 有**合理化绕过**风险。因此信息组织原则比软件设计原则更根本。


## P1 单向依赖 — 信息从抽象流向具体 `ADP, Clean Architecture`

Markdown 文件之间的引用关系形成 DAG，信息从抽象层流向具体层，不可反向引用。信息流：`theory -> rules -> schema -> SKILL.md -> procedures -> templates`。

**实践指南**：
- 严格按照依赖方向放置引用：上层定义概念和约束，下层实现和实例化
- 分发资产（rules/、skills/、knowledge/）不引用内部开发文件（docs/、.claude/）
- 违反信号：更新歧义——修改一个文件时不确定哪些文件受影响


## P2 抽象分层 — 不同抽象层级分居不同文件 `SAP, DITA`

同一个 Skill 的不同抽象层级（路由 vs 步骤 vs 理论）应分居不同文件。最常见的拆分是 SKILL.md（"做什么"）与 `*-procedures.md`（"怎么做"）。

**实践指南**：
- 每个文件包含单一主抽象层级
- `.claude-plugin/` 仅放 Manifest 元数据；所有组件放在 Plugin 根目录 `[IR-1]`
- SKILL.md 超过约 500 行时应进行拆分
- 违反信号：组件创建后"找不到"或"不生效"


## P3 稳定-易变分离 — 高频变更与低频变更解耦 `SDP`

被广泛依赖的文件（theory、rules、schema）应保持高稳定性，频繁变更的内容独立存放。

**实践指南**：
- 被 5+ 文件引用的文件月变更次数应 < 2
- 将稳定定义和易变细节拆到不同文件
- 加载策略与稳定性对齐：最稳定的文件被动加载，最易变的文件条件路由加载
- 违反信号：广泛依赖的文件频繁变更导致级联更新


## P4 信息分类 — 不同信息类型不混合 `Information Mapping, Diataxis`

一个文件只包含一种信息类型：约束放 `rules/`，概念放 `knowledge/`，步骤放 `*-procedures.md`，格式放 `_schema/`。信息类型与加载层级的完整映射见 P5 表。

**实践指南**：
- 违反信号：Claude 混淆"必须遵守的约束"和"可选的背景知识"


## P5 按需加载 — 仅加载当前上下文所需的最小信息集 `ISP, Cognitive Load`

每个文件仅在需要时加载到 LLM 上下文中，不预加载"可能用到"的内容。上下文窗口是硬约束，按需加载是"生存必需"。

| 层级 | 加载时机 | 触发条件 |
|------|---------|---------|
| `rules/` | 始终加载 | 会话开始自动加载 |
| `SKILL.md` | 触发时加载 | description 匹配或用户调用 |
| `*-procedures.md` | 匹配时加载 | SKILL.md 路由到具体子命令 |
| `knowledge/` | 引用时加载 | Skill 或 procedures 显式引用 |

**实践指南**：
- SKILL.md 放"做什么"，`*-procedures.md` 放"怎么做"
- Rules 文件精简（自动加载占基础 token 开销），knowledge 文件可详细（按需加载）
- SKILL.md 超过 ~50 行详细规则时拆出 procedures 文件
- 违反信号：Claude 在长会话中忘记早期规则、Skill 执行时丢失步骤


## P6 单一权威 — 每条信息有且仅有一个权威定义点 `DRY, SKOS`

同一信息可以在多个文件中出现，但有且仅有一个文件是权威来源。其他文件明确标注为派生。

**两种合法"反 DRY"**：
- **速查摘要**：每个文件的速查卡片是对完整内容的结构化摘要，不是独立定义
- **权威委托标记**：`<!-- 权威来源：skills/xxx/SKILL.md -->` 明确标注派生关系

**实践指南**：
- 声明权威来源 -> 文档同步路径 -> 更新时按路径级联
- 违反信号：同一概念在不同文件中定义不一致


## P7 确定性分级 — 执行力度匹配约束的关键程度 `SoC, Defense in Depth`

约束越重要，执行机制越确定性。安全关键约束不能仅靠文本规则。

**实践指南**：
- 识别"绝对不能违反"的规则 -> 实现为 PreToolUse/Stop Hook -> 文本规则作为补充说明
- 违反信号：同一条关键规则被 Claude 在不同会话中多次违反


## P8 可发现性优先 — 入口信息为发现设计，而非完整性 `Morville Honeycomb, CSO`

`description` 是 Claude 决策是否调用 Skill 的唯一输入。description 应仅包含触发条件——"何时使用"而非"做什么"。

**实践指南**：
- description 开头用 "Use when..."，只列触发关键词和场景
- description 不包含动词序列（"analyzes...generates...updates"），且 < 300 字符
- 违反信号：description 中出现"analyzes..."、"generates..."等动作描述


## P9 认知清晰 — LLM 面向指令显式且无歧义 `KISS, Prompt Engineering`

面向 LLM 的指令必须显式、精确、无歧义。模糊表述会被 LLM "合理化"偏离指令原意。

**实践指南**：
- 精确字段名和大小写：`PreToolUse` 而非 `pretooluse` `[IR-6]`
- 每条铁律附"常见绕过借口及回应"反合理化清单
- 不确定就查证，不凭记忆推断 `[IR-9]`
- 违反信号：脑海中出现"我记得这个字段是..."


## P10 契约隔离 — 数据格式由独立契约层约束 `DIP, Design by Contract`

当多个 Skill 读写同一文件时，文件格式必须由独立的 Schema 定义：`Skill A (写方) --> schema (契约) <-- Skill B (读方)`。

**实践指南**：
- Schema 文件放在独立目录（如 `knowledge/_schema/`）
- Schema 变更时必须评估所有依赖 Skill 的影响
- 违反信号：Skill A 修改输出格式后 Skill B 静默失败


## 验证回路

每次组件变更后的标准动作：

```
变更 -> 确认加载无报错 -> 手动触发目标组件确认基本行为 -> 结构检查 -> 提交
```

具体验证命令见 `common/quality.md` 验证命令库。
