# 跨组件反模式目录

> **职责**：跨组件的架构级常见错误、症状和修复方案。按需加载——故障排查时阅读。
> 组件专属反模式：AP-1,2,3（skill.md）、AP-4,8,12（hook.md）、AP-13,14（plugin.md）——位于各组件文件。

## AP-5：循环引用 `违反 P1`

- **症状**：更新时不清楚哪个文件是权威源，修改一处后另一处信息过时
- **根因**：上层文件引用下层文件的内容，下层文件又引用上层文件，形成双向依赖
- **修复**：建立单向依赖流：`theory -> rules -> schema -> SKILL.md -> procedures -> templates`。上层不引用下层的具体内容
- **检测**：绘制文件引用图，检查是否存在环路

## AP-6：急切加载 `违反 P5`

- **症状**：上下文充斥无用信息，Claude 在长会话中推理质量下降、忘记早期规则
- **根因**：会话开始时加载所有内容（理论、步骤、模板一次性全部注入）
- **修复**：按需增量加载：`rules/`（始终）-> `SKILL.md`（触发时）-> `*-procedures.md`（匹配时）-> `knowledge/`（引用时）
- **检测**：检查 rules/ 总行数是否 < 600 行；检查 SKILL.md 是否包含应按需加载的内容

## AP-7：幽灵权威 `违反 P6`

- **症状**：同一信息在多个文件中不一致（如 Skill 列表在 4 个文件中各不相同）
- **根因**：同一信息在多处独立维护，无声明哪个是权威源，更新时遗漏部分文件
- **修复**：声明权威来源，其他文件标注为派生。格式：
  - 权威文件：直接定义内容
  - 派生文件：`<!-- 权威来源：skills/xxx/SKILL.md -->`
  - 维护路径：权威 -> 派生 1 -> 派生 2
- **检测**：搜索重复出现的列表/表格，确认是否有权威声明

## AP-9：绝对路径 `违反 IR-5`

- **症状**：Plugin 在开发者机器上正常，其他用户安装后路径断裂，组件不被发现
- **根因**：`plugin.json` 或 Hook 配置中使用绝对路径或缺少 `./` 前缀
- **修复**：所有路径以 `./` 开头，Hook 脚本中用 `${CLAUDE_PLUGIN_ROOT}` 引用 Plugin 根目录
- **检测**：`grep -r "^/" plugin.json hooks/` 搜索绝对路径

## AP-10：强制设置 `违反 P9`

- **症状**：用户安装 Plugin 后需要先完成配置才能使用核心功能，大量用户在配置阶段放弃
- **根因**：核心功能要求先完成复杂配置（如定义项目结构、填写元数据模板、配置 MCP Server）
- **修复**：自然语言开箱即用（"帮我做 X" 就能开始），配置作为进阶功能渐进发现
- **检测**：测试新用户首次使用路径，确认核心功能无前置配置要求

## AP-11：混合类型文件 `违反 P4`

- **症状**：Claude 混淆约束（"必须做"）与参考信息（"可以了解"），执行时不一致
- **根因**：一个文件混合了不同信息类型——约束规则、教程说明、格式定义、理论背景
- **修复**：按信息类型拆分：
  - `rules/`：行为约束（"必须"、"禁止"）
  - `knowledge/`：概念和理论（背景参考）
  - `knowledge/_schema/`：数据格式定义（结构契约）
  - `*-procedures.md`：操作步骤（"怎么做"）
- **检测**：检查单个文件中是否混合 MUST/SHOULD 约束与概念解释

## AP-15：合理化缺口 `违反 P9`

- **症状**：Claude 创造性绕过铁律，构造听起来合理的理由（"这种情况下不适用"、"用户暗示可以跳过"）
- **根因**：铁律仅声明"不可做 X"，未附反合理化清单列举常见绕过借口
- **修复**：每条铁律配反合理化清单，格式：
  ```
  **铁律**：人工审批不可自动化
  **常见绕过借口及回应**：
  - "用户暗示同意了" -> 不算，必须明确文本确认
  - "这只是个小改动" -> 无论大小，门禁不可跳过
  - "时间紧急" -> 紧急不是降低质量的理由
  ```
- **检测**：检查每条铁律是否有配套的反合理化清单

## AP-16：不稳定基础 `违反 P3`

- **症状**：广泛依赖的文件频繁变更，每次变更触发大量下游文件级联更新
- **根因**：稳定定义（如概念模型、接口契约）和易变内容（如实现细节、版本号）耦合在同一文件
- **修复**：拆分为稳定核心（极少变更）和易变细节（允许频繁变更），下游仅依赖稳定核心
- **检测**：`git log --oneline -- <file>` 检查被广泛引用文件的变更频率

## 诊断决策树

```
问题：组件行为不符合预期
├── 组件不被发现？
│   ├── 放在 .claude-plugin/ 内？ -> AP-14 Manifest 塞满（组件移到 Plugin 根目录）
│   ├── 路径不以 ./ 开头？ -> AP-9 绝对路径
│   └── plugin.json 未同步？ -> IR-8
├── Skill 行为异常？
│   ├── description 包含行为描述？ -> AP-1 全知 Description（重写为仅触发条件）
│   ├── SKILL.md 超过 500 行？ -> AP-2 巨石 Skill（拆分 procedures）
│   └── 未记录的 frontmatter？ -> AP-3 幽灵 Frontmatter（移除非官方字段）
├── Hook 不触发？
│   ├── 事件名大小写错误？ -> AP-8 大小写盲 Hook（使用精确 PascalCase）
│   ├── 脚本无执行权限？ -> chmod +x
│   └── exit code 不是 2？ -> AP-4 纸板门禁（改用 exit 2）
├── Claude 绕过安全规则？
│   ├── 无 Hook 执行层？ -> AP-4 纸板门禁 / AP-12 纯文本安全
│   └── 有 Hook 但仍绕过？ -> AP-15 合理化缺口
├── 多文件信息不一致？ -> AP-7 幽灵权威
├── 上下文压力/忘记规则？
│   ├── rules/ 总行数 > 600？ -> AP-6 急切加载
│   └── 单文件混合多种信息类型？ -> AP-11 混合类型文件
├── 频繁级联更新？ -> AP-16 不稳定基础
├── Skill 间共享文件格式不一致？ -> AP-13 无文档契约（定义独立 Schema）
└── 用户首次使用困难？ -> AP-10 强制设置
```

## 6 步调试法

当诊断决策树未能定位问题时，使用以下系统化调试流程：

| 步骤 | 动作 | 命令 / 方法 |
|------|------|------------|
| 1 | 验证 JSON 语法 | `jq . .claude-plugin/plugin.json` |
| 2 | 检查绝对路径 | `grep -r "/Users/\|/home/" .` — 应返回空 `[IR-5]` |
| 3 | 验证执行权限 | `ls -la hooks/*.sh` — 脚本需要 `+x` 权限 |
| 4 | 独立测试组件 | 逐个启用组件，隔离问题范围 |
| 5 | 干净重装 | 卸载 -> `marketplace add` -> `install` -> 重启 |
| 6 | 查看日志 | `claude --debug` 查看详细加载和执行日志 |

**使用时机**：诊断决策树给出了方向但具体修复不明确时，按此 6 步逐一排查。
